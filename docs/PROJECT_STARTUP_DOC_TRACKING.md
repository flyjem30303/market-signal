# 專案啟動前文件製作追蹤清單（跨專案版）

本清單以「先交付、後治理」為核心原則，只做當下有價值的文件；其餘文件改在節點收斂時補。

## 1) 使用規則（CEO 運行規範）

- 只同時啟動 1 份新文件。
- 一份文件從「起草→確認→完成」走完後，再啟動下一份。
- 每份文件只要求 90% 足夠可行，不追求完美格式。
- 每個文件只聚焦一個決策輸出，避免一次收斂過多。
- 文件可用度以「是否減少明天的反覆確認」為主要標準。

### 文件深度等級

- **P0 深治理**：影響架構、資料契約、交付邊界、安全、法規、回滾機制的文件必做。
- **P1 輕治理**：需求、流程、里程碑、風險類文件可在下一節點補齊部分欄位。
- **P2 收斂治理**：格式一致性、術語對齊、跨文件關聯清理可批次做，不必阻塞起步。

## 3) 當前節點：第 1 份文件

- 目前文件：**專案啟動章程（Project Charter）**
- 這份文件要先回答 5 件事：
  1. 這個專案要解什麼問題（1 句）
  2. 誰是主要使用者（1-2 類）
  3. 一次只做什麼（Scope）
  4. 不做什麼（Out of Scope）
  5. 3 個可量化的第一階段成功指標
- 下一步指示：當你回覆「開始」，我將直接產出第一份章程草稿並與你互動微修，完成後再啟動第 2 份文件。

## 4) 每日/每輪節點更新規則

- 每輪開發後，更新 3 項欄位：  
  - 已完成項目  
  - 影響下輪的阻礙點  
  - 下一輪優先文件
- 若一輪未完成文件，僅標記「進度/阻礙」，不硬關係；保留節奏，不把文件當負擔。
- 任何 P0 文件必須在啟動當週內完成一版初稿。

## 5) 停損條款（避免過度治理）

- 同一輪最多啟動 1 個 P0 + 1 個 P1 文件。
- 若文件起草超過 45 分鐘且沒有形成決策輸出，停止文件，回到開發任務。
- 如出現新資料源、法規、權限、API 契約變更，立即插入對應 P0 文件並停止新增非關鍵文檔。
