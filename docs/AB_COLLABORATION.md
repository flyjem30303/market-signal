# A/B/C/D/E/F/I Collaboration Model

Current parallel role split is tracked in `docs/ROLE_WORKSTREAMS.md`.

本專案分成七個工作角色。

## A：PM + 開發者

責任：

- 產品規格
- 技術架構
- 前端/後端開發
- 資料庫
- 資料匯入
- 部署
- 穩定性
- SEO 技術落地
- 會員與權限系統

主要輸出：

- 可運作功能
- 技術文件
- 資料模型
- 開發任務
- 風險與成本評估

## B：行銷企劃

責任：

- 使用者定位
- SEO 策略
- 內容策略
- 週報主題
- 廣告版位策略
- 聯盟行銷策略
- 會員轉換策略
- 競品觀察
- 品牌語氣

主要輸出：

- 目標受眾
- 關鍵字策略
- 內容日曆
- 股票頁文案需求
- 轉換漏斗需求
- 廣告與聯盟版位需求

## C：專業投資顧問

責任：

- 評分模型可信度
- 指標選擇與權重建議
- 回測方法與樣本期間檢查
- 資料品質與缺值處理
- 燈號解讀是否過度武斷
- 投資人實務使用情境

主要輸出：

- 評分標準建議
- 模型權重調整需求
- 必要資料來源清單
- 回測驗證條件
- 模型風險提醒
- 專業可信度改善建議

## D：法務顧問

責任：

- 投資建議與資訊服務界線
- 免責聲明
- 廣告與聯盟行銷揭露
- 會員功能與個資風險
- 新聞內容引用與著作權風險
- 金融法規與使用者誤解風險

主要輸出：

- 法律風險提醒
- 必要揭露文字
- 禁用或需弱化的文案
- 隱私權政策與使用條款需求
- 廣告 / 聯盟揭露需求

## E：CEO

責任：

- 專案方向與產品取捨
- 長期使用價值與留存
- 商業模式排序
- 角色之間的優先順序裁決
- 判斷哪些功能會讓使用者每天或每週回來
- 回應董事長的質疑並調整策略

主要輸出：

- 產品北極星指標
- 版本優先順序
- 需砍掉或延後的功能
- 長期使用場景
- 會員與商業化節奏
- 對 A/B/C/D 的決策指令

## F：產品設計 / UIUX 設計

責任：

- 產品視覺風格
- 資訊層級與版面節奏
- 儀表板可讀性
- 手機與桌機響應式體驗
- 金融工具的專業感與信任感
- 長期使用的舒適度
- 設計系統與元件一致性

主要輸出：

- 設計原則
- 頁面資訊架構調整
- 視覺系統：色彩、字級、間距、卡片、表格、圖表
- 儀表板元件規格
- RWD 檢查清單
- 可用性問題清單

## I：雲端部署 / DevOps / 上線營運負責人

責任：

- 雲端部署架構規劃
- 正式站、測試站與預備站環境區分
- Vercel / Supabase / Cloudflare / GitHub / DNS 等服務設定指導
- 環境變數、密鑰與權限管理規則
- 網域、SSL、DNS、CDN 與部署流程
- 上線前檢查清單
- 備份、監控、錯誤追蹤與告警
- 部署回滾流程
- 指導董事長本人完成必要帳號、付款、權限與驗證操作

主要輸出：

- 雲端上線方案
- 部署與回滾 runbook
- 環境變數與密鑰管理清單
- DNS / SSL / CDN 設定檢查清單
- 上線前驗收清單
- 需要董事長本人操作的步驟清單
- 上線風險與營運成本評估

## 董事長：使用者

責任：

- 質疑 CEO 的方向與取捨
- 決定重大資源投入
- 挑戰商業模式是否合理
- 挑戰產品是否真的值得長期使用

董事長可以隨時要求 CEO 重新說明策略、調整優先順序或暫停某個方向。

## 協作原則

- A 不只做功能，必須考慮 B 的流量與商業需求。
- B 不只提想法，必須把需求轉成可驗收條件。
- B 可以主動要求 A 調整資訊架構、頁面文案、SEO metadata、內容模板、CTA、追蹤事件。
- C 可以主動要求 A 調整資料來源、評分權重、回測邏輯、模型說明與可信度揭露。
- D 可以主動要求 A 修改文案、補免責聲明、補隱私權政策、補聯盟揭露，或暫緩高法律風險功能。
- E 必須主導產品方向，確保專案不是只做一次性查詢頁，而是形成長期使用習慣。
- E 可以要求 A/B/C/D/F/I 調整優先順序。
- F 可以要求 A 調整版面、元件、資訊密度、視覺層級與互動回饋。
- F 可以要求 B 調整文案長度與 CTA 位置，避免破壞掃描效率。
- F 可以要求 C 協助判斷哪些數據應該被優先視覺化。
- F 可以要求 D 檢查揭露文字是否清楚但不干擾主要任務。
- A 可以要求 B 補充目標受眾、關鍵字、優先順序、內容範例。
- A 可以要求 C 補充指標定義、資料優先順序與可接受的模型假設。
- A 可以要求 D 補充法律風險等級與必要揭露範圍。
- A 可以要求 I 補充部署環境、權限邊界、回滾流程與上線風險。
- I 可以要求 A 補充環境變數、部署設定、健康檢查與正式環境相依服務。
- I 可以要求董事長本人完成帳號、付款、網域、OTP、身分驗證或權限授權等不可代操作事項。
- I 不得代替董事長輸入信用卡、帳號密碼、OTP、個人驗證資料或任何高敏感憑證。
- 所有 B/C/D/E/F/I 的需求必須進入文件或任務清單，不只停在聊天。

## E 何時要主動對 A/B/C/D/F/I 下決策

E 應在以下時機主動介入：

1. 專案開始偏向只做 SEO 或單次查詢時
   - 要求補上每日、每週或個人化使用場景。

2. A 新增功能但沒有留存效果時
   - 要求說明使用者為什麼會再次打開。

3. B 提出流量或變現需求時
   - 檢查是否會傷害工具信任感與長期使用。

4. C 提出模型可信度需求時
   - 判斷是否應優先於商業化。

5. D 提出法務風險時
   - 決定暫緩、調整文案或補揭露。

6. 董事長提出質疑時
   - CEO 必須回應決策理由，並視情況調整方向。

7. 產品看起來不像專業金融工具時
   - CEO 必須要求 F 介入，避免信任感不足影響留存。

8. 專案準備從本機端推進到雲端、公開網址或正式資料來源時
   - CEO 必須要求 I 介入，確認上線風險、權限邊界、回滾流程與董事長本人操作事項。

## F 何時要主動對 A 提需求

F 應在以下時機主動介入：

1. A 新增核心頁面時
   - 檢查第一屏是否能讓使用者快速理解今日狀態。

2. A 新增儀表板元件時
   - 檢查數字、圖表、標籤與說明是否有清楚層級。

3. A 新增大量文字內容時
   - 要求切成可掃描的區塊，避免金融資訊讀起來像文件堆疊。

4. A 新增行動版頁面時
   - 檢查卡片順序、按鈕尺寸、表格折疊與文字是否溢出。

5. B/C/D 提出內容、模型或法務需求時
   - F 要協助把必要資訊放進畫面，但不讓主要任務變混亂。

## B 何時要主動對 A 提需求

B 應在以下時機主動介入：

1. A 新增頁面時
   - 檢查 SEO title、description、H1、內容厚度。

2. A 新增股票頁功能時
   - 要求加入使用者會搜尋的語句與解讀文字。

3. A 新增週報頁時
   - 要求頁面有可延伸的文章結構與內部連結。

4. A 新增會員功能時
   - 要求免費/付費功能分層與轉換 CTA。

5. A 新增廣告或聯盟版位時
   - 要求不破壞信任感，並補揭露文字。

6. A 接資料來源時
   - 要求資料可支撐 SEO 內容與週報敘事。

## C 何時要主動對 A 提需求

C 應在以下時機主動介入：

1. A 新增或修改評分模型時
   - 檢查指標是否有金融意義、權重是否合理、是否需分股票與 ETF。

2. A 接資料來源時
   - 要求記錄資料來源、更新時間、缺值策略與資料品質分數。

3. A 新增回測或績效展示時
   - 要求標示樣本期間、交易成本假設、勝率、最大回檔與樣本數。

4. A 新增新聞信心評論時
   - 要求區分事實、模型推論與主觀評論。

5. A 新增週報或個股解讀時
   - 檢查是否過度保證、是否把燈號誤寫成買賣建議。

## D 何時要主動對 A 提需求

D 應在以下時機主動介入：

1. A 新增任何投資解讀、燈號、排行或週報時
   - 檢查是否需要免責聲明或弱化絕對語氣。

2. A 新增會員、收藏、Email 或個人化功能時
   - 檢查個資、隱私權政策與資料保存說明。

3. A 新增廣告或聯盟行銷時
   - 要求清楚揭露商業合作，不得偽裝成投資建議。

4. A 彙整新聞或引用第三方內容時
   - 檢查著作權、引用範圍、連結與來源標示。

5. A 準備正式上線時
   - 要求使用條款、隱私權政策、免責聲明與聯盟揭露頁完成。

## I 何時要主動對 A 或董事長提需求

I 應在以下時機主動介入：

1. A 準備部署到雲端或公開網址時
   - 要求明確列出部署平台、環境變數、建置命令、健康檢查與回滾方式。

2. A 準備連接 Supabase、真實資料來源、會員系統或金流服務時
   - 檢查權限、密鑰、正式/測試環境隔離與資料寫入風險。

3. 董事長需要開通雲端服務、付款、網域、DNS 或第三方帳號時
   - 提供逐步操作指引、必要截圖或欄位說明，但不接觸敏感憑證。

4. 專案準備正式上線時
   - 要求完成備份、監控、錯誤追蹤、告警、部署紀錄與回滾演練。

5. 任何角色提出會影響正式環境的需求時
   - 檢查是否會改變雲端成本、權限暴露、服務穩定性或資料安全風險。

## 會議節奏

目前以文件同步為主，但專案正式加入 review checkpoint 制度。

詳細制度見：

```text
docs/PROJECT_REVIEW_CHECKPOINTS.md
```

每到 checkpoint，A 暫停實作，B/C/D/E/F/I 進行深度討論。討論重點不是各自單向提需求，而是互相挑戰與建議：

- B 可以要求 C 協助把模型說法轉成使用者可理解的 SEO 內容。
- C 可以要求 B 弱化過度吸睛但不夠嚴謹的文案。
- D 可以要求 B/F 調整 CTA、揭露與版位，避免投資建議誤解。
- F 可以要求 A/B/C/D 把必要資訊重新排序，避免畫面過載。
- A 可以要求 B/C/D/F 把需求轉成可驗收條件與工程成本。
- I 可以要求 A 把部署、權限、環境變數、監控與回滾條件轉成可驗收清單。
- E 必須收斂衝突，決定下一個實作切片。

每完成一個工程切片後：

1. A 更新完成內容。
2. E 判斷是否符合長期使用工具方向。
3. F 檢查是否符合專業設計、可讀性與長期使用舒適度。
4. B 檢查是否影響 SEO、內容、轉換、變現。
5. C 檢查是否影響評分可信度、資料品質與投資解讀。
6. D 檢查是否有法律、揭露、廣告或個資風險。
7. I 檢查是否影響部署、權限、監控、成本、正式環境與回滾風險。
8. B/C/D/E/F/I 提出下一批需求。
9. A 評估成本並排入任務。

若該切片觸發 `PROJECT_REVIEW_CHECKPOINTS.md` 所列條件，A 不得直接進入下一個切片，必須先完成 checkpoint review。

## 決策規則

資源有限時，優先順序如下：

1. 能提高資料可信度的事。
2. 能降低法律與信任風險的事。
3. 能提高長期使用與回訪的事。
4. 能提高專業感、可讀性與使用舒適度的事。
5. 能降低上線、權限、密鑰、部署與回滾風險的事。
6. 能產生 SEO 頁面的事。
7. 能產生可重複內容的事。
8. 廣告與會員轉換。
9. 聯盟行銷。
