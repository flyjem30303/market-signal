# C Requests To A

這份文件記錄專業投資顧問 C 主動向 PM/開發者 A 提出的需求。

## Request 001：建立公開的評分方法論頁

提出者：C  
狀態：Done  
優先度：高

### 背景

目前網站已經有多頭健康度、回檔風險度與綜合燈號，但若要讓投資人信任，必須清楚說明分數由哪些資料與邏輯構成。

### C 的需求

請 A 建立方法論頁，至少包含：

- 六大模組與權重。
- 健康度與風險度的差異。
- 股票與 ETF 後續應分開校準。
- 回測必須揭露樣本期間、樣本數、勝率、平均報酬、最大回檔。
- 資料品質要有等級或提醒。
- 明確說明目前 mock model 不是正式投資模型。

### 驗收標準

- `/methodology` 可開啟。
- 頁面內容不把分數寫成買賣建議。
- 頁面能連回股票頁或首頁。

### A 的完成紀錄

- 已建立 `/methodology`，包含六大模組、權重、健康度與風險度差異、股票 / ETF 分開校準、回測揭露與資料品質分級。
- 頁面明確說明目前 mock model 不是正式投資模型。
- 主導覽已加入方法論入口。

## Request 002：建立模型資料品質欄位

提出者：C  
狀態：Done  
優先度：高

### 背景

正式評分必須知道資料是否完整，否則分數看起來精準但其實可能失真。

### C 的需求

請 A 在後續 schema / repository 支援：

- data_quality_score。
- stale_data_flags。
- missing_module_flags。
- model_version。
- last_updated_at。

### 驗收標準

- 文件中定義欄位。
- UI 後續可顯示「資料完整度」。

### A 的完成紀錄

- 已在 schema、Supabase migration 與資料文件中加入資料品質欄位。
- `SignalSnapshot` 已加入 data quality 欄位。
- 今日燈號頁已顯示資料品質等級、完整度與模型版本。
